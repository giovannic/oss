00003  *
00004  *\section section1 A. OVERVIEW
00005  *Two major pieces of 3rd-party software provide much of the infrastructure
00006  * for FutureTool:
00007  *
00008  *    DPCL: http://www-124.ibm.com/developerworks/opensource/dpcl
00009  *
00010  *    Dyninst: http://www.dyninst.org
00011  *
00012  *    Only a single DPCL SuperDaemon runs on each 'node' in a cluster. When the first
00013  *    SuperDaemon starts, it obtains a lock and holds that lock as long as it is running.
00014  *    Additional DPCL user 'connect' requests to the same 'node' result in additional
00015  *    DPCL SuperDaemons being created. These additional SuperDaemons attempt to obtain
00016  *    the same lock, but cannot, and they transfer the 'connection' to the first DPCL
00017  *    SuperDaemon and exit.
00018  *
00019  *    The DPCL SuperDaemon invokes a single DPCL 'communication daemon' for each user-id
00020  *    running on a cluster 'node'. Each DPCL 'user-id' has a single 'communication daemon'.
00021  *    Part of the DPCL user 'connection' processing is to have shared memory segments
00022  *    properly established and shared between the 'communication daemon' and a user's
00023  *    application (target process).  Architecture-specific 'locking' code prevents the
00024  *    'communication daemon' and target processes from simultaneously updating shared
00025  *    structures in the shared memory segments.
00026  *
00027  *    FUTURETOOL never suspends waiting for a user's DPCL 'communication' daemon to 'connect'.
00028  *
00029  *
