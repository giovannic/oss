Introduction

 

The steps to use a working copy of Open|SpeedShop on targeted platforms where static linking is required are described below.  Since there are some O/S features not available on the compute nodes of the platforms being addressed here, the Open|SpeedShop tool must be used in an alternative manner than it is on general vanilla flavored clusters.  Open|SpeedShop is built for running the analysis and viewing on the front end nodes and for the backend nodes the collectors to gather the performance data are built to be inserted into the application.  The Open|SpeedShop build on the front end node creates GUI and CLI tools for viewing the performance data gathered on the compute nodes.  The second build step builds the targeted runtime libraries and the targeted performance information gathering collectors.  These runtime libraries and collectors will be linked into the user's application and do require the application to be relinked, but not necessarily recompiled.  Open|SpeedShop provides a command, “osslink” that can be embedded into the users makefile to make the inclusion of the Open|SpeedShop runtimes and collectors easier.  The details are described in the sections below.


Where to get help?

This version of Open|SpeedShop is under development but is usable to get program counter sampling information at a minimum.  If you decide to try it, please contact the Open|SpeedShop extended team by emailing questions to: oss-questions@openspeedshop.org or post questions to the newly created Open|SpeedShop Forum.




Using the Open|SpeedShop on Cray-{XT, XE,XK} compute node runtimes and collectors to gather application performance data

 

There is a script, "osslink", that is installed into the <openspeedshop_install_dir>/cray-{xt4,xt5,xe,xk}/bin directory that can be used in the user's makefile to infuse the Open|SpeedShop runtimes and collector into the user's application without recompilation.  Here is an example:

 

default: $(TARGET)

 

$(TARGET): $(OBJ)

$(F90) -o $@ $(FFLAGS) $(OBJ) $(LDFLAGS)

 

oss-pcsamp: $(OBJ)

osslink -v -c pcsamp $(F90) -o $(TARGET)-pcsamp $(FFLAGS) $(OBJ) $(LDFLAGS)

 

In this example, the oss-pcsamp target uses the "osslink" script to create another executable, $(TARGET)-pcsamp that includes the Open|SpeedShop libraries and Program Counter Sampling collector code that will gather performance information about the $(TARGET) application.  Then you run your code as you would normally do.   So, the process is to find the key point where  your application makefile does the final link of all the pieces and add another target that uses osslink.

 

NOTE: KEY POINTS

Once, you have created the new executable, run this executable as you would the one without the Open|SpeedShop collection code inserted, except for two changes:

 

          a) use -bb on aprun (aprun -bb)

          b) Set the environment variable: OPENSS_RAWDATA_DIR to a shared file system path.  This is where Open|SpeedShop will write the raw data files containing the performance data gathered while the application is running.

 

After the run is completed, run the Open|SpeedShop utility tool that converts the raw data files into an Open|SpeedShop database file.   The command would be:

        ossutil $OPENSS_RAWDATA_DIR


The command could actually be included in the batch script as a convenience to eliminate another manual step. 


Viewing the Open|SpeedShop database file (.openss suffix)


After the ossutil command is run and the raw performance data files are converted into an Open|SpeedShop database file, we are ready to view the data with the Open|SpeedShop base tool using one of these commands.

 

      a) Using the GUI:

                 openss -f <Open|SpeedShop database file>

      b) Using the CLI:

                 openss -cli -f <Open|SpeedShop database file>

 

                 > expview    - shows the performance data information

                 > expstatus – shows the metadata information
