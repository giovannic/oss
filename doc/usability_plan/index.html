Section 1.     Introduction

The primary goal of the plan for Open|SpeedShop usability testing is to create a highly usable performance tool product. SGI suggests a phased usability testing strategy allowing the Open|SpeedShop tools to be tested for usability as they are developed.  This document covers usability in the usage of the Open|SpeedShop tools and also in the ability to extend the tools.   Extending the tool refers to the ability of the user to add new performance experiments and new GUI or CLI views of the data collected via the new experiments.

1.1 Open|SpeedShop Project Information

 

The Open Source Parallel Performance Measurement Package Statement of Work document dated May 27, 2004 contains the technology and strategy to be used by SGI to deliver a parallel performance measurement package.  SGI will create a performance analysis tool for the Intel IA64, IA32, EM64T platforms and AMD AMD64 platform, running under the Linux operating system.

SGI will release, primarily under the GPL and LGPL Open Source licenses, the baseline functionality as described in section 2.1 of the above mentioned SOW and will productize an enhanced Pro version.  Support for single system image (SSI) machines, support for clusters (i.e., multiple OS kernels), exclusive and inclusive user time, program counter (PC) sampling, MPI call tracing, input/output tracing, floating point exception tracing and CPU hardware performance counter experiments are the key components of the baseline functionality.  The performance tool will be designed in such a manner that will allow users to easily extend the tool by adding their own experiments.

Another key feature of the performance tool is its usability.  The user interface will be designed for scientists not computer scientists.  To make the tool usable for a greater range of users, the performance tool will provide novice user Wizards, easily understood language in the user interface components, and context sensitive help.  These features and additional features that comprise the baseline version are described in detail in section 2.1 of the SOW.

The design and structure of this project allows for the performance tool described in this document to be ported to several other platforms through a moderate effort.  Since the performance tool will use dynamic instrumentation as one of the primary building blocks, platforms supported by the Dyninst dynamic instrumentation package are ready candidates for porting.

The performance tool will use dynamic probe class library (DPCL) as a building block component for cluster system support.  The use of DPCL will provide a portable means for the performance tool to supply cluster support for platforms supported by the Dyninst dynamic instruction package.  DPCL uses the Dyninst application programming interface (API) to provide its instrumentation capability.

Availability of the new performance tool based on SpeedShop technology on Linux platforms will enable Fortran (77, 90, and 95), C, and C++ programmers to use an advanced performance analysis tool within the Open Source environment.

Section 2.     Goal of the Usability Plan

The goal of the usability plan is to outline a plan for usability testing and analysis on the Open|SpeedShop tools that have user interfaces. In addition to the tool user interfaces, the installation/configuration, the interfaces of the plugins (API) and the scripting API for the CLI will also have usability reviews. Schedules for the testing and for incorporation of the usability testing feedback are defined.

The goal of usability testing is to find out what is and is not working well with the user interface. In a usability test, you usually want to answer questions like these:

  • Do users complete a task successfully?
  • If so, how fast do they do each task?
  • Is that fast enough to satisfy them?
  • What paths do they take in trying?
  • Do those paths seem efficient enough to them?
  • Where do they stumble?  What problems do they have?  Where do they get confused?
  • What words or paths are they looking for that are not now on the user interface?

The usability studies will initially be done at the SGI facilities in either Eagan or Mt.View.  At a mutually agreed upon point in the Open|SpeedShop development, usability tests will be done at one or more of the ASC laboratories.

Section 3.     Open|SpeedShop Tools subject to the Usability Plan

The tools that will be studied are the performance tools graphical user interface (GUI), the interactive command line (CLI), and the batch command. These are the three Open|SpeedShop tools that have user interfaces. In addition to the tool user interfaces, the installation/configuration, the interfaces of the plugins (API) and the CLI scripting API will also have usability reviews.  Each usability item has a specific usability plan defined in the sections that follow.


Section 4.     What are the General Open|SpeedShop Features to be evaluated?

Each Open|SpeedShop tool has its own unique usability features to be evaluated, but in general the goal is to test ease of use and how quickly the user can effectively use the tools. In the paragraphs below each tools unique features are discussed. These unique features define the usability study specific to the tool.  The Plugin API, the installation/configuration of the tools, and the CLI scripting API will be evaluated.

4.1 GUI Evaluation Features

The GUI will be a more complicated evaluation because of the many facets of a graphical user interface. First time users� and power users� ease of use are very important to the success of the tool. Several tasks can be created that would test the ease of use for those two categories of users. These tasks would generally consist of:

  • Loading files, getting source, simple wizard usage.
  • Working example of single process job.
  • Launching a new experiment.
  • Loading existing performance data.
  • Working example of multiple process jobs and creation of uses own experiment.
  • Attach to executable.
  • Wizards full usage

4.2 Interactive CLI Tool Evaluation Features

In the first usability study, the interactive CLI tools primary evaluation items will be the syntax of its commands and the arguments to those commands. Participants in the study will determine if the command and arguments are easily understood and if they supply the necessary functionality. In the second usability study, participants will actually be given tasks to accomplish and then be given an evaluation sheet to record their experience in using the tool.  These tasks would generally consist of:

  • Loading files, getting source, simple wizard usage.
  • Working example of single process job.
  • Launching a new experiment.
  • Loading existing performance data.
  • Working example of multiple process jobs and creation of user's own experiment.
  • Attach to executable.

4.3  Batch Command Tool Evaluation Features

In the first usability study, the batch commands evaluation items will be the command syntax and options used in the batch command. Participants in the study will determine if the command and arguments are easily understood and supply the necessary functionality. In the second usability study, participants will actually be given tasks to accomplish and then be given an evaluation sheet to record their experience using the tool.

4.4 Scripting API Evaluation Features

In this usability study, the scripting API of the CLI tool evaluation items will be the syntax of the scripting API and the arguments to those routines. Participants in the study will determine if the routines and arguments are easily understood and if they supply the necessary functionality.

4.5 Tool Installation and Configuration Evaluation Features

In this usability study, the installation and configuration of the Open|SpeedShop tool evaluation items will be the ease of configuration and installation. Participants in the study will determine if the installation can be done in a straightforward and understandable manner with as little human intervention as possible.   The study will also evaluate how much system knowledge is needed to install Open|SpeedShop.

Another evaluation feature is the configuration of the Open|SpeedShop tool.  Questions related to the configuration of the tool are:

  • Can the tool be configured easily for the various cluster and single system image (SSI) machines that the Open|SpeedShop performance tool targets?
  • How difficult is it to install the tool in a non-default location?
  • If there are two installations of the tool, one in the default location and one in a non-default location, can both versions work at the same time?

4.6 Plugin API Evaluation Features

In this usability study, the creation and inclusion of Open|SpeedShop experiment specific plugins are the Open|SpeedShop tool evaluation items that will be the evaluation items. Participants in the study will determine if the plugin creation is well documented and can be done with reasonable effort. 

The questions to be answered for this usability study are:

  • Can a user take the provided documentation for adding plugins to the tool and create and add the plugin to the Open|SpeedShop tool?  
  • How easy is it to install the new plugins?

Section 5.     Plan for GUI Usability Testing

SGI recommends that we have a phased usability strategy so that as the new user interfaces are designed and developed they are also tested for usability. The Open Source Parallel Performance Measurement Package Statement of Work document dated May 27 identifies three phases for usability studies: one in the first three months and then two others three months and six months after the first. It may be that we will want to do at least one more for the multiprocessing and cluster support portion of the GUI.  This usability study can be done when the multiprocessing GUI view designs and/or prototypes are available.  In this version of the plan, the multiprocessing related usability study is scheduled to be executed in phase 3.

5.1 Phase 1 GUI Usability Study: Basic Support

SGI will set up a canned set of common tasks the candidate needs to complete (in 20 minutes or less). The initial tasks, representing basic support, would be something like:

  • Run a default performance experiment on an executable.
  • Load the executable into Open|SpeedShop.
  • Run the experiment on the executable.
  • Determine which routine took the most time.
  • Which source file was that routine from?

 

This first phase of usability will test the base functionality. Can a user figure out how to invoke the tool and set up a predefined performance experiment? Can the user get a report back with the information they are looking to find? This would address basic usability, such as, screen management, menu management, basic GUI gestures, help feedback, context sensitive help, initial learning curve, and work flow.

5.2 Phase 2 GUI Usability Study: Run experiment and obtain reports

Phase 2 of the GUI usability study will test the following tasks.  Can a user run a simple application under the tool and get live data back? Do the reports we provide give useful and meaningful feedback? This would be more of a usability study of reports, e.g. is the information we display meaningful? An additional goal would be to see if the user could save the data to a file and read that file in and get useful information from loading an existing data file.

This study would also address such items as report layout, report data, additional GUI gestures. It would also, again, test basic workflow management.

5.3 Phase 3 GUI Usability Study: Create Own Experiment and Multiprocess Management

Phase 3 of the GUI usability study will test the following tasks. Can a user do the tasks in the usability studies defined by phase 1 and phase 2? Additionally, can the user create their own experiment from a list of collectors? Can the user get a meaningful report back? Can the user load a multiprocessor job and not be overwhelmed and confused by the volume of information returned to the user.

This third phase of usability testing would also test the scalability regarding multiprocess management of processes and performance data. Also, performance measurement reports and performance monitoring workflow would be studied.

5.4 GUI Usability Study - Who, Where, and When

SGI and ASCI need to define the individuals that will be participants in the usability studies for the Open|SpeedShop GUI tool. An appropriate number of participants for a usability study of the GUI tool might be three or four people.

The phases of the GUI Usability study will be done at these locations and dates:

  • Phase 2 completed on 04/27/05 at LANL.  Follow up information on the changes to the GUI are included in the Milestone 4 version of the GUI Usability Phase 2 Result document.   See the comments, questions, and responses here.
  • Phase 3 on (TBD) at SGI in Mt. View or Eagan or an ASC lab

 

Section 6.     Plan for Interactive Command Line Interface (CLI) Usability Testing

SGI recommends that we have a phased usability strategy for evaluation of the interactive command line processor. The first phase would be to review the commands that are proposed for the interactive command line tool and give feedback to SGI on whether or not the commands are easily understood and the command syntax is acceptable. A scenario of how the commands would be used will be part of the Command Line Interface: Command Definition Document.

6.1 Phase 1 CLI Usability Study: Command Syntax

Participants will determine if the commands and command arguments are easily understood and supply the necessary functionality. The participants would review the command scenarios that would be provided to make some initial judgments on usability. This user feedback will be given to the developers of the CLI tool, who can then make the necessary adjustments to the commands and the command arguments, if any are needed.

6.2 Phase 2 CLI Usability Study: Run experiment and obtain reports

In the second usability study, participants will actually be given tasks to accomplish and then be given an evaluation sheet to record their experience in using the tool. This will be very similar to the tasks used for the GUI phase 2 usability study, except that the tasks will be completed using only the command line interface.

SGI will set up a canned set of common tasks the candidate needs to complete (in 20 minutes or less). The initial tasks, representing basic support, would be something like:

  • Run a default performance experiment on an executable.
  • Load the executable into Open|SpeedShop.
  • Run the experiment on the executable.
  • Determine which routine took the most time.
  • Which source file was that routine from?

 

This second phase of the CLI usability study will test the base functionality. The types of actions that will be tested are listed in the following sentences.  Can a user figure out how to invoke the tool and set up a predefined performance experiment? Can the user get a report back with the information they are looking to find?  This would address basic usability, such as, help feedback, initial learning curve, and work flow.  Can a user run a simple application under the tool and get live data back? Do the reports we provide give useful and meaningful feedback? This would be more of a usability study of reports, e.g. is the information we display meaningful?  An additional goal would be to see if the user could save the data to a file and read that file in and get useful information from loading an existing data file.

This would also address such items as report layout, report data, and basic workflow.

6.3 CLI Usability Study - Who, Where, When

SGI and ASCI need to define the individuals that will be participants in the usability studies for the Open|SpeedShop command line interface tool. An appropriate number of participants for a usability study of the CLI tool might be three or four people.

The phases of the Interactive Command Line Interface Usability study will be done at these locations and dates:

  • Phase 1 on 10/25/04 over email and a meeting to discuss the emailed CLI Command Syntax.   Follow up changes to the CLI  Command Syntax document is included in the Milestone 3 version of the CLI documentation.   See the comments, questions, and responses here.
  • Phase 2 on 07/28/05 was conduct at LANL.  Follow up information on the changes to the CLI are included in the Milestone 4 version of the CLI Usability Phase 2 Result document.   See the comments, questions, and responses here.

Section 7.     Plan for Batch Command Tool Usability Testing

SGI proposes that ASC and SGI have a phased batch command tool usability strategy so that as the new user interfaces are designed and developed they are also tested for usability. The Open Source Parallel Performance Measurement Package Statement of Work document dated May 27 does not identify any usability study for the batch command, however SGI recommends a minimum of two should be done.

Phase 1 of the batch command tool usability study would be a review of the batch command syntax and its arguments. Phase 2 of the study would be actual usage of the batch tool with user feedback given back to the developers of the batch tool.

7.1 Phase 1 Batch Command Usability Study: Command and Command Arguments

Participants in the usability study will determine if the batch command and batch command arguments are easily understood and supply the necessary functionality. The participants would review the batch command scenarios that would be provided, to make some initial judgments on usability. This user feedback will be given to the developers of the batch tool, who can then make the necessary adjustments to the batch command arguments, if any are needed.

7.2 Phase 2 Batch Command Usability Study: Run experiment and obtain reports

In the second usability study, participants will actually be given tasks to accomplish and then be given an evaluation sheet to record their experience in using the tool. This will be very similar to the tasks used for the GUI phase 2 usability study. However, the participants would have to understand how to set up a batch job to accomplish the previously interactive tasks.

SGI will set up a canned set of common tasks the candidate needs to complete (in 20 minutes or less). The initial tasks, representing basic batch command usage, would be something like:

  • Run a default performance experiment on an executable.
  • Load the executable.
  • Run the experiment on the executable.
  • Determine which routine took the most time.
  • Which source file was that routine from?

 

This second phase of batch command tool usability study will test the base functionality. Tasks that will be tested are: Can a user figure out how to invoke the tool and run a predefined performance experiment? Can the user get a report back with the information they are looking to find? These tasks would address basic usability, such as, command setup, initial learning curve, and work flow.

Can a user run a simple application under the batch tool and get performance measurement data back is the main usability item tested in Phase 2.

7.3 Batch Command Usability Study - Who, Where, and When

SGI and ASCI need to define the individuals that will be participants in the usability studies for the Open|SpeedShop batch command tool. An appropriate number of participants for a usability study of the batch command tool might be three or four people.

The phases of the Batch Command Tool Usability study will be done at these locations and dates:

  • Phase 1 December of 2004 over email or web posting of the document with email comments or a meeting to discuss the findings
